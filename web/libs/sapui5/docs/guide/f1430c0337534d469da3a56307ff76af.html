UI Adaptation at Runtime: Enable Your App

Here's what you have to consider when developing apps that support UI adaptation at runtime (RTA).

RTA is a feature of SAPUI5 flexibility services that allows end users to personalize object pages when running an app in the SAP Fiori launchpad and key users without technical knowledge to easily make UI changes for all users of an app. For more information, see SAPUI5 Flexibility Services: Adapting UIs Made Easy.

Note

If your app is based on SAP Fiori elements, it automatically supports RTA. You only have to consider the following if you use extension points to modify it, for example to add an additional view. You can adapt the UI of apps based on SAP Fiori elements using the UI Adaptation editor in SAP Web IDE. For more information, see UI Adaptation at Runtime in the SAP Cloud Platform Web IDE guide.

Tip

When viewing a sample in the SAPUI5 Demo Kit, you can check whether a control is enabled for UI adaptation at runtime by choosing the wrench symbol (Adapt UI) at the top right. If UI elements of the sample get highlighted when you hover over or select them, it's an indicator that the control is enabled for UI adaptation at runtime. (You can also directly in the sample try out the UI adaptation features.)

Very important: Use stable IDs

As stable IDs are used to identify the controls that a key user can adapt using RTA, you have to use stable IDs for each control used in the supported controls and in each view of your app.

We recommend that you also use stable IDs for all other controls and views of your app, to ensure that they can be adapted in future using RTA. For more information, see Stable IDs: All You Need to Know.

Caution

Never change the IDs when creating the next version of your app if it was already delivered. Otherwise you'd risk inconsistencies and the loss of UI changes that a key user might have done to the delivered version.

Shall the key user be able to add additional UI elements?

To enable the key user to make visible additional properties of an OData entity in RTA (typically when the key user adds fields or custom fields), you have to do the following:

  • Use data binding in the container where this shall be possible.

  • Make sure that the related OData model is the default model of your app component. For more information, see Assigning the Model to the UI.

Use asynchronous loading of views to improve performance

To enable processing of UI changes directly on XML views, use the asynchronous loading of views. For more information, see Instantiating Views.

Something isn’t working like it should? Please check the troubleshooting information. For more information, see Troubleshooting.

Note

Bear in mind that depending on the controls used, it might be necessary that key users make their UI changes in RTA in edit mode as well as in display mode of the app. Exception: smart forms show changes made in one mode automatically in the other.

For more information about how RTA works for key users and how it gets enabled in the SAP Fiori launchpad, search for Adapting SAP Fiori UIs at Runtime and Enabling UI Adaptation at Runtime in the documentation for your SAP NetWeaver version on the SAP Help Portal.
