Tree Table

The sap.ui.table.TreeTable control is a table that includes a hierarchy. It is based on the sap.ui.table.Table control.

API Reference

For more information, see the API Reference for sap.ui.table.TreeTable and sap.ui.model.odata.v2.ODataTreeBinding.

Overview

Table controls simply set the binding contexts for all rows every time a change in the visual representation is detected. Most of the time, this happens due to a scroll operation by the user.

For TreeTables, a common case is the expand operation. If a hierarchy is modifiable, data binding contexts are also reset when the hierarchy is changed, for example, if nodes are removed or added.

With the aid of ODataListBinding and ODataTreeBinding, a paging mechanism only requests the data entries needed for the currently visualized snapshot of the full data set. Therefore, only a limited and minimal excerpt of the original data set is available on the client.

This in turn means that all user operations, as well as programmatic API-calls, might trigger back-end requests to get the full data set.

Features
Note

The features are only available when using sap.ui.model.odata.v2.ODataModel.

ODataTreeBinding implements the most important set of features that are needed to operate on a tree structure in the back end:

  • Expanding and collapsing of nodes

  • Paging big data sets

  • Lazy loading child nodes on expansion of the parent node

  • Selections:

    • Adding and removing selection ranges

    • Clearing the selection

    • "Select all"

  • Hierarchy Maintenance:

    • Adding and removing nodes

    • Submitting the changed data back to the service

  • Different operation modes

1) Filtering and searching with the minimal annotation set is only possible with application filters. For more information, see sap.ui.model.odata.v2.ODataTreeBinding.filter. The application filters will be sent with each data request. Control filters are not supported in this case, as the paging and lazy loading mechanisms have a technical limitation on what can be done on the client. Complex OData tree filtering can only be done in the back end, and can only be achieved when using the hierarchy-node-descendant-count-for annotation.
2) For services with the hierarchy-node-descendant-count-for annotation, the operation modes Client and Auto will still fall back to the same client-side logic as with the minimal annotation set. This means that hierarchy maintenance is not yet possible for those cases.
3) You should never use the Auto-Expand feature for the operation mode Server and an OData service with the minimal annotation set, even though it is supported, because this has a huge impact on performance.

The expansion of a tree to a certain level requires a series of sequential back-end requests, which is not feasible for big data sets and applications that contain a huge amount of data. We recommend to use the hierarchy-node-descendant-count-for annotation set. The Auto-Expand feature is fully functional and supported for this annotation without any performance impact.
